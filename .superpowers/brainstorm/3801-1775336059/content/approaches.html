Abordagens para o Export Pipeline

Escolha a arquitetura que melhor se encaixa no projeto

A

html-to-image direto no componente

Cada slide wrapper em App.tsx recebe um ref. A função de export chama toPng(ref.current) diretamente, sem abstração. Simples e direto.

Prós

  • Mínimo de código novo
  • Sem camadas extras
  • Fácil de entender

Contras

  • Lógica de export espalhada em App.tsx
  • Difícil de reusar em outros contextos

Recomendado para: MVP rápido

B

useExport hook + SlideExportControls

Hook useExport(ref, { filename }) encapsula toda lógica: estado de loading, chamada ao html-to-image, download, toast. Componente SlideExportControls consome o hook e renderiza os botões.

Prós

  • Separação clara de responsabilidades
  • Hook testável isoladamente
  • App.tsx fica limpo
  • Reutilizável se o projeto crescer

Contras

  • Mais arquivos para criar

⭐ Recomendado — melhor trade-off

C

Feature dedicada: features/export/

Feature completa com ExportProvider (contexto global), fila de jobs, progress bar por slide, e orquestração centralizada de todos os exports.

Prós

  • Escalável para batch de muitos slides
  • Estado global compartilhado

Contras

  • Over-engineering para o uso atual
  • Muito mais código que o necessário
  • Contexto global desnecessário agora

Recomendado para: app com muitos carrosséis simultâneos
